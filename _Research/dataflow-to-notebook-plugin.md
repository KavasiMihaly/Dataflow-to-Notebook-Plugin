**Date:** 2026-05-02
**Author:** Research pass for OneDayBI marketplace plugin
**Status:** Research only ‚Äî no implementation yet
**Target marketplace:** [OneDayBI-Marketplace](https://github.com/KavasiMihaly/AI-plugins) (companion to existing `dbt-pipeline-toolkit`)

## 1. Goal

Package the existing **Dataflow Gen1 ‚Üí Fabric Medallion Notebooks** workflow as a single installable Claude Code plugin distributed via the OneDayBI marketplace. The end-to-end experience the user should get after `/plugin install`:

> "Drop me into an empty repo, point me at a Power BI workspace ID, run my orchestrator. I'll export every Gen1 dataflow, parse the M code, scaffold a Fabric medallion project, generate bronze + silver PySpark `.ipynb` notebooks, deploy and validate them in Fabric, and produce a migration report."

The plugin will fold together capabilities that today live as **standalone skills**, **standalone agents**, and **manual scripts** in `C:\Users\kavas\Documents\GitHub\Fabric - Dataflow to Notebook` ‚Äî and ship them as one install with a documented orchestration flow.

## 2. What already exists (and what we're packaging)

### 2.1 Existing reference implementation

`C:\Users\kavas\Documents\GitHub\Fabric - Dataflow to Notebook` is a **completed migration project** that:

- Exported **8 Dataflow Gen1 definitions** from a Power BI workspace via `Export-PowerBIDataflow`
- Parsed JSON ‚Üí 37 `.pq` files via `extract_m_from_json.py`
- Analyzed each query: source type, role (output_entity / staging / transformation / helper), conversion risk
- Produced bronze + silver `.ipynb` notebooks (10 bronze, 6 silver) using `fabric-bronze-builder` and `fabric-silver-builder` agents
- Deployed via `fab` CLI / REST API and validated row counts in lakehouses

This is a **proven workflow** ‚Äî the plugin's job is to package it, not invent it.

### 2.3 Gaps not covered by any existing asset

The current workflow has **manual stages** that should become plugin-orchestrated:

1. **Dataflow analysis / dependency map** ‚Äî currently hand-written in `_Research/Dataflow Analysis and Dependency Map.md`. Should be auto-generated by a new `m-query-analyst` agent (or a data-explorer specialization).
2. **Medallion mapping** ‚Äî currently hand-written in `_Documentation/Medallion Layer Design.md`. Should be auto-derived: each .pq query gets classified as bronze (output entity, raw read) / silver (transformation) / skip (helper) based on the existing `extract_m_from_json.py` role classifier.
3. **M-code conversion risk assessment** ‚Äî currently hand-written in `_Research/M Code Conversion Risk Assessment.md`. Should be auto-produced by scanning .pq files for known risk patterns (Excel.Workbook, custom M functions, `AzureStorage.Blobs`, etc.). The risk patterns are already documented from the reference project.
4. **No orchestrator** ‚Äî today the workflow is a sequence of agent + skill calls a human chains together. The dbt plugin proves a single "drop me in and go" orchestrator is feasible.
5. **No validation pass** ‚Äî there's no `fabric-pipeline-validator` analogue to `dbt-pipeline-validator`. Should validate `.ipynb` JSON shape, lakehouse bindings, `read_bronze()`-only contract for silver, row count assertions, and end-to-end Fabric run.
6. **No deployment skill orchestration** ‚Äî `deploy_all_notebooks.py` exists in the reference project's Architecture Setup folder, but it's project-internal not plugin-shipped.

## 3. Why a plugin (and why now)

### 3.1 Marketplace fit

The existing `dbt-pipeline-toolkit` plugin ships a CSV-to-star-schema-on-SQL-Server experience. A Fabric-medallion-from-Dataflow-Gen1 plugin is the **direct Fabric companion** to it, with the same overall philosophy (orchestrator + specialists + skills + validation hooks) and the same target audience (Power BI / Fabric data engineers). The marketplace is currently single-plugin; the new plugin doubles its catalog and gives the OneDayBI brand a Fabric-specific entry.

### 3.2 Microsoft's deprecation timeline creates urgency

Per Microsoft's April 2026 guidance, **Dataflow Gen1 is now officially Legacy** ([Microsoft blog](https://powerbi.microsoft.com/en-us/blog/dataflows-thank-you-for-eight-years-of-gen1-and-why-gen2-is-the-future/)). Existing Gen1 dataflows still run, but no new feature work will land. Microsoft offers three official Gen1 ‚Üí Gen2 paths (export template, copy/paste, Save As ‚Äî see [Move queries from Dataflow Gen1 to Dataflow Gen2](https://learn.microsoft.com/fabric/data-factory/move-dataflow-gen1-to-dataflow-gen2)) but none target **notebooks** as the destination. For organizations whose target architecture is medallion lakehouse + Fabric notebooks (the recommended pattern in [Fabric medallion architecture docs](https://learn.microsoft.com/fabric/onelake/onelake-medallion-lakehouse-architecture)), there is no first-party migration path.

This is the gap our plugin fills. There is **no widely-known open-source M-to-PySpark converter** ([web search confirmed](https://www.codeconvert.ai/sql-to-pyspark-converter) ‚Äî only SQL‚ÜíPySpark tools exist, not M‚ÜíPySpark).

### 3.3 Multi-agent orchestration is a strong fit for this workflow

The migration is naturally multi-stage with distinct expertise per stage:

- Extraction (PowerShell + JSON parsing) ‚Äî mechanical
- Analysis (M-pattern recognition, risk assessment) ‚Äî needs LLM judgment
- Project scaffolding ‚Äî interactive, deterministic
- Bronze notebook generation ‚Äî pattern-following, parallelizable
- Silver notebook generation ‚Äî judgment + reference to bronze contract
- Deployment + validation ‚Äî mechanical with assertion checks

Mapping these to specialized subagents is the same pattern that worked for the dbt plugin. The reference project's existing agents (`fabric-bronze-builder`, `fabric-silver-builder`) already follow this pattern.

## 5. Proposed orchestration flow (13 stages)

Modeled on the dbt-pipeline-orchestrator's 13-stage flow, adapted for Fabric. **Two interactive touchpoints**, everything else autonomous.

| Stage | Owner | Auto/Interactive | Output |
|---|---|---|---|
| 0 | Orchestrator | Auto | Detect mode: fresh build vs. incremental (existing `0 - Architecture Setup/`) |
| 1 | Orchestrator + user | **Interactive** | Ask: workspace ID, target path, lakehouse names ‚Äî store in `migration-design.md` |
| 2 | `dataflow-gen1-extractor` skill | **Interactive (PowerShell auth)** | Generate `Export-AllDataflows.ps1`, ask user to run it; parse JSON ‚Üí .pq files + manifest CSV |
| 3 | `m-query-analyst` agent | Auto | Classify each .pq: source type, role, complexity; produce dependency map |
| 4 | `m-query-analyst` agent | Auto | Risk assessment scan (12 known patterns); produce `M Code Conversion Risk Assessment.md` |
| 5 | `business-analyst` agent | Auto | Draft medallion mapping (bronze/silver/skip per query); section 8 = semantic layer plan |
| 6 | Orchestrator | **Interactive (plan mode)** | Present medallion mapping + risk summary; user approves or revises |
| 7 | `fabric-project-setup` agent | Auto | Run `fabric-project-initializer` skill; scaffold folders, project-config.yml, CLAUDE.md |
| 8 | `fabric-bronze-builder` (parallel fan-out, 1 per output_entity) | Auto | Generate `nb_bronze_*.ipynb`; m-to-pyspark-converter drafts the read logic |
| 9 | `fabric-silver-builder` (sequential, then parallel by independent dataflow) | Auto | Generate `nb_silver_*.ipynb`; consume bronze contract via `read_bronze()`-only |
| 10 | `fabric-notebook-deployer` skill | Auto | `fab import` each notebook into the target workspace |
| 11 | `fabric-cli-runner` | Auto | `fab job run` bronze notebooks first, then silver ‚Äî capture run-ids |
| 12 | `fabric-pipeline-validator` agent | Auto | `fabric-lakehouse-reader` queries each target table; assert row counts; produce `Validation Results.md` |
| 13 | Orchestrator | Auto | Write final `Migration Report.md` with stats, deviations, deploy log |

### 5.1 Strict conformance gate (Stages 8 + 9)

Same pattern as dbt plugin: each builder returns a JSON envelope with `conforms_to_plan`, `deviations[]`, `errors[]`. Orchestrator halts on any deviation and asks user accept-deviation vs. abort. Critical for Fabric because:

- Silver builder's "read from bronze only" rule is a hard contract. If a builder slips a `spark.read.csv(...)` in (because the M code did it directly), the medallion architecture quietly breaks. The validator will catch it but at gate 9 we should catch it at the seam.
- M-to-PySpark conversion produces TODO markers for unknown functions. If a builder ships TODOs unresolved, the notebook fails at runtime. Gate forces resolution before deployment.

### 5.2 Section 8 = Semantic Output Plan

Borrowed directly from the dbt plugin's design: section 8 of `migration-design.md` is the user-facing answer to "what tables will exist when you're done?" with a Mermaid diagram of bronze‚Üísilver‚Üígold dependencies. The Stage 6 plan-mode gate leads with this so the user is approving outcomes, not just file shapes.

## 6. Critical lessons from the dbt plugin to apply here

The dbt plugin's `_Documentation/plugin_learnings.md` contains 10 hard-won findings ‚Äî **every single one applies to this plugin**. Summary of how each affects the design:

### 6.1 Finding 1 ‚Äî 3-part agent namespace (subdirectory layout)

Plugin agents under subdirectories register as `<plugin>:<subdir>:<frontmatter-name>`. Every reference site needs the full 3-part name:

```yaml
# In orchestrator's tools allowlist
tools: Agent(fabric-dataflow-migration-toolkit:fabric-bronze-builder:fabric-bronze-builder, ...)

# In every Task spawn
subagent_type: "fabric-dataflow-migration-toolkit:fabric-bronze-builder:fabric-bronze-builder"

# In invocation example
claude --agent fabric-dataflow-migration-toolkit:fabric-migration-orchestrator:fabric-migration-orchestrator "Migrate dataflows"
```

**Action:** keep subdirectory name === frontmatter name field for every agent. Test on a fresh install before shipping; the dev-mode `--plugin-dir ./` path uses bare names and hides the namespace mismatch.

### 6.2 Finding 2 ‚Äî Plugin agents can't declare `permissionMode`/`hooks`/`mcpServers` in frontmatter

The existing `fabric-bronze-builder/agent.md` has `permissionMode: acceptEdits` and `isolation: worktree` in its frontmatter. **`permissionMode` will be silently stripped at install time** (the `isolation: worktree` field is supported and stays). The orchestrator must pass `mode: "acceptEdits"` at the call site for every Task spawn instead.

**Action:** strip `permissionMode:` from every plugin-shipped agent's frontmatter during the vendor step. Add it as an explicit `mode:` arg in the orchestrator's `Task(...)` calls.

### 6.3 Finding 3 ‚Äî Background subagents stall on permission prompts

Critical for Stages 8, 9, 10, 11, 12 ‚Äî every notebook write, every `fab` deployment, every `fab job run`, every Python-script call. Without the plugin-level Bash auto-approval hook (Finding 9), these stall silently in background mode.

**Action:** mirror `approve-plugin-bash.py` from the dbt plugin and extend the allowlist with Fabric-specific patterns:

```python
# Plugin-internal scripts (same shape as dbt plugin)
r'^python "?\$\{CLAUDE_PLUGIN_ROOT\}/skills/[a-z0-9_-]+/scripts/[a-z0-9_-]+\.py"? .*'

# Fabric CLI
r'^fab (auth|import|export|job|workspace|item|cd|ls|get) .*'
r'^pip install ms-fabric-cli$'

# PowerShell for the export step (user-run, but might be called for status)
r'^pwsh -File ".*Export-AllDataflows\.ps1"$'

# Notebook conversion utilities
r'^python ".*convert_py_to_ipynb\.py" .*'

# Plus the dbt-equivalent git/find/ls atoms
```

### 6.4 Finding 4 ‚Äî Orchestrator only works as main-thread `--agent`

Document loudly in README:

```bash
# CORRECT ‚Äî main-thread invocation:
claude --agent fabric-dataflow-migration-toolkit:fabric-migration-orchestrator:fabric-migration-orchestrator "Migrate Power BI dataflows from workspace 4353fefd... into Fabric medallion notebooks"

# INCORRECT ‚Äî auto-delegation from a session:
"@fabric-migration-orchestrator migrate my dataflows"   # becomes a leaf subagent, can't spawn workers
```

### 6.5 Finding 5 ‚Äî `userConfig` env vars are namespaced

Every plugin-internal Python script must remap `CLAUDE_PLUGIN_OPTION_<KEY>` ‚Üí `<KEY>`. Apply the same `_load_plugin_userconfig_env()` helper to every Python script that reads environment variables (Fabric workspace ID, lakehouse names, Azure auth):

```python
def _load_plugin_userconfig_env():
    keys = (
        'FABRIC_WORKSPACE_ID', 'FABRIC_WORKSPACE_NAME',
        'BRONZE_LAKEHOUSE', 'SILVER_LAKEHOUSE', 'GOLD_LAKEHOUSE',
        'AZURE_TENANT_ID', 'AZURE_CLIENT_ID', 'AZURE_CLIENT_SECRET',
    )
    for key in keys:
        if not os.environ.get(key):
            fallback = os.environ.get(f'CLAUDE_PLUGIN_OPTION_{key.lower()}')
            if fallback:
                os.environ[key] = fallback
```

**Open question (Problem B in dbt findings):** the dbt plugin observed that `userConfig` prompts didn't fire on install. We don't yet know the root cause. Mitigation: ship a README "Manual config" section showing how to edit `settings.json` directly, plus a SessionStart hook that detects unset Fabric env vars and prints a setup message.

### 6.6 Finding 7 ‚Äî `${CLAUDE_PLUGIN_ROOT}`, not `$HOME/.claude/skills/`

Every existing `fabric-*` agent and skill file uses `C:\Users\kavas\.claude\skills\<name>\scripts\<file>.py` paths (Windows-style, hardcoded user). On a fresh install, these paths don't exist.

**Action:** during the vendor step, do a mechanical replace across every agent.md and SKILL.md:

- `C:\Users\kavas\.claude\skills\` ‚Üí `${CLAUDE_PLUGIN_ROOT}/skills/`
- Backslashes ‚Üí forward slashes inside the script-path segment
- The `fabric-bronze-builder` README reference example currently shows `python "C:\Users\kavas\.claude\skills\fabric-cli-runner\scripts\run_fabric_cli.py" import ...` ‚Äî this becomes `python "${CLAUDE_PLUGIN_ROOT}/skills/fabric-cli-runner/scripts/run_fabric_cli.py" import ...`

### 6.7 Finding 8 ‚Äî Skill preload is 2-part namespace

Every plugin agent's `skills:` frontmatter must use 2-part `<plugin>:<skill>` names:

```yaml
# Before (current standalone form)
skills: fabric-cli-runner, fabric-lakehouse-reader, m-to-pyspark-converter

# After (plugin form)
skills: fabric-dataflow-migration-toolkit:fabric-cli-runner, fabric-dataflow-migration-toolkit:fabric-lakehouse-reader, fabric-dataflow-migration-toolkit:m-to-pyspark-converter
```

### 6.8 Finding 9 ‚Äî Atomic Bash commands only

Every Bash invocation in agent bodies, skill SKILL.mds, hook commands, and orchestrator stage prompts must be **single atomic operations**. No `&&`, `||`, `;`, `|`, no subshells, no heredocs to native executables.

The reference project's `Export-AllDataflows.ps1` runs a complete export in one PowerShell call ‚Äî that's fine because it's atomic from Bash's perspective. But things like:

```bash
# WRONG (compound)
fab import nb.Notebook -i nb.ipynb && fab job run nb.Notebook | tee log.txt

# RIGHT (atomic ‚Äî separate tool calls)
# Call 1
fab import nb.Notebook -i nb.ipynb
# Call 2 (after reading exit code in LLM text)
fab job run nb.Notebook
```

**Action:** every stage prompt in the orchestrator must produce atomic-only Bash. The `m-query-analyst` and `fabric-pipeline-validator` agents must follow the same rule. Inline the rule in every specialist's agent body (per Finding 9 Round 3 ‚Äî belt-and-suspenders inlining across orchestrator, specialists, project CLAUDE.md, and global CLAUDE.md).

### 6.9 Finding 10 ‚Äî Dev-mode hides install-mode bugs

Every fix above is invisible until you install on a fresh machine via the marketplace. **Action:** before shipping v0.1, install on a clean Windows machine that has never seen the repo, run the orchestrator end-to-end with a real workspace ID, verify every stage produces side effects (not just exit codes).

## 8. Risks and open questions

### 8.1 Plugin distribution risks (carried over from dbt plugin)

1. **Subagent delegation on plugin agents may regress.** Per Finding 6 of the dbt learnings, this feature is technically "Not Planned" per Anthropic's issue tracker (#19276 closed 2026-02-27) but works empirically. We should have a fallback architecture documented (skill orchestrator + general-purpose fan-out) before shipping.
2. **`userConfig` may not prompt on install.** The dbt plugin observed this and the root cause is unknown. Mitigation: ship a README setup-from-scratch path that doesn't depend on the prompt firing.
3. **`${CLAUDE_PLUGIN_ROOT}` substitution in markdown body is officially documented but not 100% empirically verified for agent bodies.** The dbt plugin assumes it works (it does in `plugin.json`); next fresh install validates it. If the substitution doesn't reach agent body, fallback is a SessionStart hook exporting `PLUGIN_ROOT` to shell env.

### 8.2 Fabric-specific risks

1. **Fabric CLI auth has known quirks.** The reference project's `_Research/fab-auth-login-browser-issue.md` documents browser auth failures on Windows. The plugin's `fabric-cli-runner` skill needs to surface auth errors clearly and document the service-principal fallback path.
2. **Notebook deployment via REST API is finicky.** The reference project has `_Research/fab-notebook-deployment-research.md` documenting the difference between `.py` and `.ipynb` deployment. **`.ipynb` is mandatory** ‚Äî `.py` notebooks deploy as a single mega-cell. The validator hook should reject `.py` files in `3 - Notebooks/`.
3. **Power Query M conversion produces TODOs.** The `m-to-pyspark-converter` SKILL.md explicitly says complex M expressions become TODO markers. The orchestrator must surface these to the user (not silently ship a notebook with `# TODO: convert this manually`). Stage 8 conformance gate enforces resolution.
4. **Excel.Workbook source has no native PySpark equivalent.** RISK-03 in the existing risk catalog. v0.1 should ship a documented mitigation (pandas + openpyxl in-cell or pre-converted CSV); the converter must produce a clear TODO with the recommended option.
5. **OneLake shortcuts vs. direct ABFSS reads.** The reference project's RISK-01 maps `AzureStorage.Blobs(...)` to `spark.read.csv("abfss://...")`. This requires either a OneLake shortcut to be configured or workspace identity / linked service to be set up out-of-band. The plugin should detect this requirement and raise it to the user during the plan-approval stage, not silently emit code that 401s at runtime.

### 8.3 Orchestrator design open questions

1. **Should Stage 2 (PowerShell export) be foreground-only?** Today the user runs `Export-AllDataflows.ps1` manually because of interactive auth. Could we offload to a service principal? Probably yes, but the auth setup may be more friction than the manual run. Default to manual; document SP path.
2. **Parallel vs. sequential silver builds?** Bronze can run fully in parallel (one per output_entity). Silver has implicit dependencies on bronze tables existing ‚Äî but builders only need `read_bronze("name")` to compile, not to run. Suggested: parallel silver build (file generation), then sequential `fab job run` deployment. Same pattern as dbt plugin's parallel dim build then sequential fact deployment.
3. **What's the canary stage?** dbt plugin runs first staging model sequentially as a canary. Fabric equivalent: deploy + run the simplest bronze notebook first; if it fails, abort before fanning out. The reference project doesn't do this; we should add it.
4. **Where does TMDL/semantic model handoff live?** Out of v0.1 scope, but the dbt plugin has `pbip-from-dbt` skill for this. A future `pbip-from-fabric` skill could plug in at Stage 13.

### 8.4 Marketplace integration

1. The OneDayBI marketplace's `marketplace.json` needs a new entry pointing at the new repo. Same shape as the dbt entry. No technical risk.
2. Plugin name collisions to watch: `fabric-dataflow-migration-toolkit` is wordy but distinct from `dbt-pipeline-toolkit`. Alternatives considered: `fabric-medallion-toolkit` (broader, less Gen1-specific ‚Äî better if we later expand to non-Gen1 sources), `dataflow-to-notebook` (matches the user's working folder name but ambiguous between Gen1 and Gen2). **Recommendation: `fabric-medallion-toolkit`** ‚Äî leaves room for v0.2 Gen2 source, gold layer, and other medallion-targeted features without renaming.
